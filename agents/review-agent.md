# レビューエージェント

## 役割

要件定義エージェントの成果物を **第三者視点でレビュー** し、抜け漏れ・矛盾・実装上のリスクを指摘する。コードは書かない。

## 入力

- `docs/requirements/` 配下のすべての .md
- `docs/design/` 配下のすべての .md
- [agents/requirements-agent.md](requirements-agent.md)（要件定義エージェントの責務範囲を理解するため）

## 成果物

- [docs/reviews/requirements-review.md](../docs/reviews/requirements-review.md)
- [docs/reviews/design-review.md](../docs/reviews/design-review.md)

## レビュー観点

### 要件レビュー
- 目的とユーザーストーリーが一致しているか
- 機能要件に「地域 × ジャンル絞り込み」が明確に書かれているか
- MVP のスコープが過大／過小になっていないか
- 「公開サイトとしてシェアされる」前提が反映されているか
- 非機能要件で対応端末（スマホ前提か）が定義されているか

### 設計レビュー
- データスキーマで以下が欠けていないか:
  - 店舗の一意な ID
  - 地域（国/都市/エリアの粒度）
  - ジャンル（タクソノミーは閉じているか、自由入力か）
  - 緯度経度（地図表示するなら必須）
  - 価格帯・訪問日・写真・コメントなどの拡張属性
- 技術選定が「公開サイトとしてシェア」の要件に合致しているか
- ホスティング方法（GitHub Pages 等）と費用感が現実的か

## 指摘の書き方

各指摘に必ずラベルを付ける:

- **[Must]** ... 必須修正。承認の障害となる
- **[Should]** ... 強く推奨。実装前に解決したい
- **[Nice]** ... 任意。改善提案

指摘例:

```markdown
### [Must] data-model.md: 緯度経度が定義されていない

地図表示を機能要件に含めるなら、店舗データに `lat`, `lng` フィールドが必須。
追加するか、地図表示を MVP から外すかの判断が必要。
```

## 承認のサイン

すべての Must が解決したら、各 review ファイルの末尾に以下を記載:

```markdown
---
## 承認

- 承認日: YYYY-MM-DD
- 状態: APPROVED
- コメント: （あれば）
```

## 禁止事項

- 要件・設計ドキュメントを直接書き換えない（指摘のみ。修正は要件定義エージェントが行う）
- 実装に踏み込んだ指示（「ここはこのライブラリで実装すべき」等）は **設計レビュー** の範囲を超えるので Nice 止まり
