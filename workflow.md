# エージェント連携フロー

## エージェント体制

### コア開発チーム（4 エージェント）

```
┌──────────────────────────────────────────────────────┐
│         開発計画管理エージェント（横断）              │
│  進捗管理 / 意思決定の記録 / 未解決事項の追跡         │
│  → docs/plan/                                         │
└──────────────────────────────────────────────────────┘
                          │
        ┌─────────────────┼─────────────────┐
        ▼                 ▼                 ▼
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│ 要件定義AG    │→ │ レビューAG    │→ │ 実装AG        │
│ docs/        │  │ docs/reviews/│  │ src/, data/  │
│ requirements,│  │              │  │              │
│ design       │  │              │  │              │
└──────────────┘  └──────────────┘  └──────────────┘
        ↑                  │
        └─── 差し戻し ──────┘
            （Must 指摘あり）
```

### キュレーションサブチーム（3 エージェント、D-020）

```
[ユーザー] →「店を増やしたい」要望
      ↓
┌────────────────────┐
│ キュレーション計画AG │  → ブリーフ（target / 件数 / source 順）
└────────────────────┘
      ↓
┌────────────────────┐
│ 店舗リサーチAG      │  → 候補リスト（schema 準拠 JSON + 要検証フラグ）
└────────────────────┘
      ↓
┌────────────────────┐
│ キュレーション統合AG│  → Sheets 貼り付け用 TSV + 確認事項 + 却下リスト
└────────────────────┘
      ↓
[ユーザー] → Sheets に貼る → Run workflow → サイト反映
```

サブチームの成果物は **`docs/curation/curation-log.md` の R-NNN ラウンド** に集約。
スキーマ拡張やタクソノミー追加が必要な場合は、統合 AG が **要件定義AG にエスカレーション** する。

## フェーズ別の流れ

### フェーズ 0: プロジェクト構造設計

担当: **開発計画管理エージェント**（+ ユーザー）

1. フォルダ構造を設計
2. 各エージェントの指示書を `agents/` に配置
3. `docs/plan/project-plan.md` でマイルストーンを定義

### フェーズ 1: 要件定義

担当: **要件定義エージェント**
監督: **開発計画管理エージェント**

1. `docs/requirements/` の 4 ファイルを記入
2. `docs/design/` の 3 ファイルを記入
3. 重要な意思決定は **開発計画管理エージェント** が `docs/plan/decisions-log.md` に記録
4. 未確定事項は `docs/plan/open-issues.md` に登録

完了したらレビューエージェントに引き継ぐ。

### フェーズ 2: レビュー

担当: **レビューエージェント**
監督: **開発計画管理エージェント**

1. `docs/reviews/requirements-review.md` を記入
2. `docs/reviews/design-review.md` を記入
3. 各指摘に `[Must]` / `[Should]` / `[Nice]` を付与
4. **開発計画管理エージェント** が指摘事項を `open-issues.md` に集約
5. **Must** が残っていれば要件定義エージェントに差し戻し
6. すべて解決したら、レビューファイルの末尾に **APPROVED** を記載

### フェーズ 3: 実装

担当: **実装エージェント**
監督: **開発計画管理エージェント**

1. 承認済み要件・設計に従って `src/` を構築
2. `data/restaurants.json` にサンプルデータを投入
3. ローカルで動作確認
4. 実装中に判断が必要な事項は **開発計画管理エージェント** に報告 → `decisions-log.md` に記録

### フェーズ 4: 公開・運用

担当: **実装エージェント**（+ ユーザー）
監督: **開発計画管理エージェント**

1. Vercel にデプロイ
2. Google Sheets との連携を確認
3. GitHub Actions の定期再ビルドを設定
4. `project-plan.md` を「完了」に更新

## 受け渡しのルール

- **一方向に流す**: 後続フェーズのエージェントは前フェーズの成果物を書き換えない。修正が必要なら担当エージェントに差し戻す。
- **変更履歴を残す**: 各ドキュメントの末尾に `## 更新履歴` セクションを設け、日付と変更内容を記載。
- **未確定事項は明示**: 不明点・保留事項は `> TODO:` プレフィックスで残し、**開発計画管理エージェント** が `open-issues.md` で追跡。
- **意思決定は記録**: 「なぜそうしたか」は **開発計画管理エージェント** が `decisions-log.md` に記録。

## エスカレーションのルール

以下が発生したらユーザーに即報告:
- `Blocker` ラベルの Issue が発生した
- フェーズの DoD が満たせないことが判明した
- スコープ外の機能要望が出てきた（→ MVP に入れるか将来拡張か判断）
